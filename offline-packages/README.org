#+title: Offline Emacs Toolkit
#+author: James Dyer
#+options: toc:2 num:nil
#+startup: content

A self-contained toolkit for bringing up my Emacs configuration on machines with no network access. The package set is keyed purely on Emacs version — architecture is not tracked separately. Three scripts do the work:

- =create-install.sh= — on an /online/ machine: render an init.el from a per-version package list, download every package into an isolated =$HOME=, tar it up as an ELPA mirror, and (by default) chain into =build-toolkit.sh= to produce the deployable tarball.
- =build-toolkit.sh= — on any /online/ machine with this repo: bundle a chosen version's mirror with =init.el=, the =Emacs-vanilla= + =Emacs-DIYer= literate configs, an installer, and (optionally) the GNU Emacs source tarball.
- =setup.sh= — lives inside each built toolkit, run on the /offline/ target to drop everything into =~/.emacs.d/=.

The rendered =init.el= auto-detects =~/elpa-mirror-emacs-*/= at startup and points =package-archives= at the local mirror when one is present. No manual toggle required.

* Directory layout

#+begin_example
offline-packages/
├── create-install.sh            ← step 1: build a mirror for an Emacs version
├── build-mirror.sh              ← called by create-install.sh; wraps elpa-mirror
├── build-toolkit.sh             ← step 2: assemble a deployable toolkit
├── fetch-source.sh              ← download GNU Emacs source into sources/
├── init.el.in                   ← template rendered per Emacs version
├── packages/
│   ├── emacs-27.2.el            ← one package symbol per line + SOURCE_VERSION header
│   └── emacs-30.2.el
├── local-packages/              ← drop zone for self-written / locally-patched packages
│   ├── foo.el                   ← single-file package, OR
│   └── bar/bar.el               ← multi-file package in its own subdir
├── starters/                    ← optional example init snippets
│   ├── emacs-27.2.el            ← installed as init-starter.el on the target
│   ├── emacs-30.2.el
│   └── coding.el                ← language-agnostic coding companion
├── tools/                       ← drop zone for non-Emacs binaries (gitignored)
│   └── jdtls/…                  ← e.g. language servers, debug adapters
├── docs/                        ← drop zone for reference docs (gitignored)
│   └── setting-up-emacs-for-coding/
├── sources/                     ← GNU Emacs source tarballs (gitignored)
│   ├── LATEST_STABLE            ← single line, e.g. "30.1"
│   └── emacs-30.1.tar.xz        ← fetched by fetch-source.sh
└── README.org                   ← this file
#+end_example

Mirrors are /build artefacts/, not source, so they're written outside the repo by default:

#+begin_example
$OFFLINE_MIRRORS_DIR/                                    ← default: ~/.cache/emacs-offline-toolkit/mirrors/
├── emacs-27.2/
│   ├── elpa-mirror-*.tar.gz                             ← produced by build-mirror.sh
│   └── init.el                                          ← rendered from init.el.in
└── emacs-30.2/…
#+end_example

Override the location by exporting =OFFLINE_MIRRORS_DIR= (e.g. to keep mirrors on a separate disk) — both =create-install.sh= and =build-toolkit.sh= honour it.

Current targets:

#+begin_src sh :results output :exports results
./build-toolkit.sh --list
#+end_src

* Package configuration format

Each =packages/emacs-<VER>.el= is a plain list — one package symbol per line, blank lines and =;= comments ignored.

#+begin_src text
async
use-package
yaml-mode
doom-themes
…
#+end_src

Edit this file to change the package set — nothing else needs editing. Missing/incompatible packages log to =*Messages*= rather than aborting, so leaving =ef-themes= in an Emacs 27.2 list is harmless (it just won't install).

The GNU Emacs source tarball bundled with the toolkit is decoupled from the mirror version: =build-toolkit.sh --with-source auto= reads =sources/LATEST_STABLE= (a single canonical version used for /every/ target). See "Pre-fetching GNU Emacs source" below.

* Local packages (self-written / locally-patched)

Anything dropped under =offline-packages/local-packages/= is bundled into every toolkit regardless of target version — these are packages you've written yourself or pulled from GitHub and tweaked, i.e. things that don't live in an ELPA archive.

Two supported shapes:

#+begin_example
local-packages/
├── my-thing.el          ← single-file package
└── dape/
    └── dape.el          ← multi-file package; the subdir joins load-path
#+end_example

=build-toolkit.sh= copies this directory into the toolkit tarball; =setup.sh= installs it to =~/.emacs.d/local-packages/= on the offline target (backing up any existing one with the usual =.pre-toolkit-<STAMP>= suffix). No byte-compilation is done at install time — sources are dropped verbatim and the target Emacs compiles on first load.

The rendered =init.el= puts =~/.emacs.d/local-packages/= plus every immediate subdirectory onto =load-path= before =package-initialize=, so a =(require 'foo)= or =use-package foo= (without =:ensure=) just works. Nothing is auto-=require=d — you still declare usage from your own init.

* Starter init snippets

Each =starters/emacs-<VER>.el=, if present, is bundled as =init-starter.el= at the root of the toolkit tarball, and =setup.sh= installs it to =~/.emacs.d/init-starter.el= on the target. It is /not/ auto-loaded — the rendered =init.el= copy-over is left untouched.

The snippets are distilled from the author's live config: =use-package= blocks for the themes, =diff-hl=, =selected-window-accent-mode=, =simply-annotate=, =ztree=, =dape= (30.x), =ollama-buddy= (30.x), etc. They assume the corresponding package is already in the offline mirror (true by construction, since the snippet is matched to =packages/emacs-<VER>.el=).

To opt in on the target, append one line to =~/.emacs.d/init.el=:

#+begin_src emacs-lisp
(load (expand-file-name "init-starter" user-emacs-directory) t t)
#+end_src

…or cherry-pick blocks into your own init. To disable: just remove the =load= line (or don't add it).

Adding a starter for a new version: drop =starters/emacs-<VER>.el= next to =init.el.in= — =build-toolkit.sh= picks it up automatically by filename match on =EMACS_VERSION=.

** Coding companion (=starters/coding.el=)

=starters/coding.el= is a version-agnostic snippet distilled from =docs/setting-up-emacs-for-coding/README.org= — eglot hooks for Java/C#/C++/TypeScript, flymake =M-n=/=M-p=, eldoc, corfu with =corfu-auto nil=, dape =F5=/=F9=/=F10=/=F11=, =cmake-mode=, =typescript-mode=, =highlight-indent-guides=. It's bundled as =coding-starter.el= at the toolkit root and installed to =~/.emacs.d/coding-starter.el= (not auto-loaded). Opt in alongside or instead of the per-version starter:

#+begin_src emacs-lisp
(load (expand-file-name "coding-starter" user-emacs-directory) t t)
#+end_src

Blocks that need Emacs 29+ (eglot, =global-corfu-mode=) are guarded by =fboundp= checks, so loading under 27.2/28.x is a no-op for those features rather than an error. The JDTLS entry looks for =~/.emacs.d/bin/jdtls/bin/jdtls= — populate =tools/= (see below) to wire it up.

* Tools drop-zone (non-Emacs binaries)

=offline-packages/tools/= is a drop zone for /non-Emacs/ binaries shipped alongside the toolkit: language servers (JDTLS, csharp-ls, clangd, ada_language_server), debug adapters (=netcoredbg=), formatters. The contents are not tracked in git (gitignored, with a =.gitkeep= for structure) because they're platform-specific and often large.

Layout mirrors whatever you want under =~/.emacs.d/bin/= on the target:

#+begin_example
tools/
├── jdtls/
│   ├── bin/jdtls
│   ├── plugins/…
│   └── com.microsoft.java.debug.plugin-0.53.2.jar
└── netcoredbg
#+end_example

=build-toolkit.sh= copies =tools/= into the toolkit (preserving executable bits); =setup.sh= installs it to =~/.emacs.d/bin/= on the target with the usual =.pre-toolkit-<STAMP>= backup. The coding companion starter looks for =~/.emacs.d/bin/jdtls/bin/jdtls= automatically.

Add =~/.emacs.d/bin= to =PATH=, or reference binaries by absolute path from your own =eglot-server-programs=.

* Docs drop-zone

=offline-packages/docs/= ships reference material (cheat sheets, coding guides) so an offline target has the same docs the online machine has. Gitignored like =tools/= / =local-packages/=. =build-toolkit.sh= bundles it; =setup.sh= installs to =~/.emacs.d/docs/=. Nothing is loaded from it automatically — it's purely reading material.

The upstream =setting-up-emacs-for-coding= guide lives here by convention; see =docs/setting-up-emacs-for-coding/README.org= for the stack the coding companion starter implements.

* Building a mirror (step 1)

Run on the online machine:

#+begin_src sh
cd ~/.emacs.d/offline-packages
./create-install.sh 27.2               # mirror + deployable toolkit (default)
./create-install.sh 27.2 --mirror-only # stop after the mirror
./create-install.sh 27.2 --out-dir /tmp # toolkit tarball goes to /tmp
#+end_src

By default, once the mirror is built =create-install.sh= invokes =build-toolkit.sh --target emacs-<VER>=, producing =$HOME/.emacs.d/emacs-offline-toolkit-<VER>-*.tar.xz= /without/ the GNU Emacs source tarball bundled. Pass =--with-source auto= (or an explicit version) to include the source, or =--mirror-only= to stop after the ELPA mirror under =$OFFLINE_MIRRORS_DIR/= (default =~/.cache/emacs-offline-toolkit/mirrors/=).

=create-install.sh= will:

1. Look for an =emacs-<VER>= binary under =$HOME/emacs-versions/=, =$HOME/bin/=, or =$PATH=. If none is found, it offers to invoke =Emacs-vanilla/scripts/build-emacs-versions.sh <VER>= to build it.
2. Read =packages/emacs-<VER>.el= (package list).
3. Render =init.el.in= → a staged init.el with packages injected.
4. Run =HOME=<staging> emacs-<VER> --batch -l <rendered-init.el>= so packages land in an isolated ELPA directory, independent of your real =~/.emacs.d/=.
5. Call =build-mirror.sh= (with =EMACS= and =HOME= pointing at the staged Emacs/ELPA) to tar up the mirror into =$OFFLINE_MIRRORS_DIR/emacs-<VER>/= (default =~/.cache/emacs-offline-toolkit/mirrors/emacs-<VER>/=), and copy the rendered =init.el= alongside the tarball.
6. Unless =--mirror-only= was passed, chain into =build-toolkit.sh --target emacs-<VER>=. Add =--with-source auto= (or =--with-source VER=) to also bundle the GNU Emacs source tarball.

Mirror tarball name:

#+begin_example
$OFFLINE_MIRRORS_DIR/emacs-<VER>/elpa-mirror-emacs-<VER>-<OS>-<ARCH>-<STAMP>-<N>pkgs.tar.gz
#+end_example

OS/arch stay in the filename for provenance, but the directory is flat — you can build on different distros and all the tarballs coexist under =$OFFLINE_MIRRORS_DIR/emacs-<VER>/= (newest wins at toolkit-build time).

Inside the tarball:

| File                 | Purpose                                                   |
|----------------------+-----------------------------------------------------------|
| =archive-contents=   | ELPA index Emacs uses for =package-install=               |
| package =.tar=/=.el= | One artefact per installed package                        |
| =MANIFEST.txt=       | Build provenance (host, kernel, Emacs version, timestamp) |
| =PACKAGES.txt=       | Sorted =package  version  summary= listing                |

** Running build-mirror.sh standalone

If you already have packages installed in your /real/ =~/.emacs.d/elpa= and just want to tar them up, you can call =build-mirror.sh= directly:

#+begin_src sh
./build-mirror.sh                 # writes under $HOME by default
./build-mirror.sh /path/to/out    # explicit output directory
EMACS=emacs-27.2 ./build-mirror.sh
#+end_src

Output layout: =<OUT_DIR>/emacs-<VER>/<mirror-base>.tar.gz= (flat).

* Building a deployable toolkit (step 2)

Interactive:

#+begin_src sh
./build-toolkit.sh                      # fzf / numbered menu picker
#+end_src

Non-interactive:

#+begin_src sh
./build-toolkit.sh --target emacs-27.2
./build-toolkit.sh --target emacs-30.2 --out-dir /tmp
#+end_src

With a bundled GNU Emacs source tarball (version read from =sources/LATEST_STABLE=):

#+begin_src sh
./build-toolkit.sh -t emacs-27.2 --with-source auto
#+end_src

Or with an explicit source version:

#+begin_src sh
./build-toolkit.sh -t emacs-27.2 --with-source 27.2
#+end_src

Flags:

| Flag                       | Purpose                                          |
|----------------------------+--------------------------------------------------|
| =-t=, =--target SPEC=      | =emacs-VER= spec; skip picker                    |
| =-o=, =--out-dir DIR=      | Output directory (default: =$HOME/.emacs.d=)     |
| =-s=, =--with-source VER=  | Bundle =emacs-VER.tar.xz=; =auto= reads LATEST_STABLE |
| =--gzip=                   | Use gzip (.tar.gz) instead of xz -9e (.tar.xz)   |
| =--no-smoke-test=          | Skip the post-staging "boot init.el" check       |
| =-l=, =--list=             | List available targets and exit                  |
| =-h=, =--help=             | Show help                                        |

** Smoke test

Before compressing, =build-toolkit.sh= extracts the bundled mirror into a throwaway =$HOME=, locates =emacs-<VER>= (checking =$HOME/emacs-versions/emacs-<VER>/bin/emacs=, =$HOME/bin/emacs-<VER>=, =$PATH=), and runs it in =--batch= mode against the staged =init.el= with =(kill-emacs 0)=. If the Emacs binary can't be found for the target version the test is /skipped/ with a warning; if it runs and the init fails to load, the build aborts before the tarball is produced. Pass =--no-smoke-test= to bypass.

** What goes into the toolkit

#+begin_example
emacs-offline-toolkit-<VER>-<OS>-<ARCH>-<STAMP>/
├── MANIFEST.txt            ← build provenance + file inventory
├── PACKAGES.txt            ← sorted name/version/summary of every bundled pkg
├── README.org              ← this document
├── init.el                 ← per-version rendered init
├── early-init.el           ← if present in ~/.emacs.d/
├── setup.sh                ← installer for the offline machine
├── rollback.sh             ← reverse the most recent setup.sh install
├── elpa-mirror-*.tar.gz    ← the selected ELPA mirror
├── Emacs-vanilla/          ← literate config (READMEs, init.el, scripts)
├── Emacs-DIYer/            ← literate config (READMEs, init.el, dictionary)
├── local-packages/         ← self-written / locally-patched packages (if any)
├── tools/                  ← non-Emacs binaries (language servers, debug adapters)
├── docs/                   ← reference docs (coding guide, cheat sheets)
├── init-starter.el         ← optional starter snippet (not auto-loaded)
├── coding-starter.el       ← optional coding companion (not auto-loaded)
└── emacs-<VER>.tar.xz      ← optional GNU Emacs source
#+end_example

=Emacs-vanilla= and =Emacs-DIYer= are pulled fresh from the =main= branches of =captainflasmr/Emacs-vanilla= and =captainflasmr/Emacs-DIYer= on GitHub. Pass =--local-configs= to copy from =~/.emacs.d/<d>/= instead (for testing uncommitted edits).

Excluded from =Emacs-vanilla/= and =Emacs-DIYer/= to keep the toolkit small:

- =.git=, =.gitignore=, =.gitmodules=, =.github=, =.claude=
- =*.elc=, =*~=, =.#*=, =#*#=
- =eln-cache=, =auto-save-list=

** Size

Default compression is =xz -9e= (extreme preset). Pass =--gzip= for faster compress/decompress at the cost of a larger tarball — useful during iteration. Typical sizes with xz:

| Target               | Mirror | Configs | Total (no source) | Total (+source) |
|----------------------+--------+---------+-------------------+-----------------|
| Emacs 27.2 (14 pkgs) | ~600 KB | ~700 KB | ~1–2 MB          | ~55–65 MB       |
| Emacs 30.2 (28 pkgs) | ~1.7 MB | ~700 KB | ~2–3 MB          | ~55–65 MB       |

The =tools/= drop-zone (language servers, debug adapters) dominates the total — ~420 MB uncompressed, ~150 MB compressed. On a full bundle the tarball lands around ~150–200 MB.

** --no-tools update tarballs

Once the target has the big initial install, subsequent rebuilds rarely need to re-ship =tools/= — language servers and debug adapters move slowly. Pass =--no-tools= to =create-install.sh= or =build-toolkit.sh= to produce an "update" tarball that omits the drop-zone:

#+begin_src sh
./create-install.sh 30.2 --no-tools
#+end_src

The resulting filename gets a =-notools= suffix for disambiguation, and MANIFEST.txt notes the skip. On the target, =setup.sh= guards the tools install with =if [[ -d tools ]]=, so an absent =tools/= leaves the existing =~/.emacs.d/bin/= untouched — package churn ships, binaries stay put. Typical size drop: ~150 MB → ~3 MB (xz).

* Deploying on the offline target (step 3)

Copy the =.tar.xz= to the target by whatever means (USB, air-gapped transfer, …), then:

#+begin_src sh
tar -xJf emacs-offline-toolkit-*.tar.xz
cd emacs-offline-toolkit-*/
./setup.sh
#+end_src

=setup.sh= will:

1. Back up any existing =~/.emacs.d/{init.el,early-init.el,Emacs-vanilla,Emacs-DIYer,local-packages,bin,docs,init-starter.el,coding-starter.el,elpa}= and any =~/elpa-mirror-emacs-*/= with a timestamped =.pre-toolkit-<STAMP>= suffix. The =elpa/= move forces the next launch to reinstall every package from the freshly extracted mirror (not from stale =.elc/.eln= left over from a previous toolkit).
2. Extract the ELPA mirror tarball into =~/=, creating =~/elpa-mirror-emacs-<VER>-<OS>-<ARCH>-<STAMP>-<N>pkgs/=.
3. Install the literate config directories under =~/.emacs.d/=.
4. Install =init.el= plus any bundled =early-init.el= / =init-starter.el= / =coding-starter.el= / =local-packages/= / =tools/= (→ =~/.emacs.d/bin/=) / =docs/= (→ =~/.emacs.d/docs/=).
5. Report instructions for building the bundled Emacs source, if present.

To undo: run =./rollback.sh= from inside the same extracted toolkit dir (or from anywhere — it operates on =$HOME= / =$HOME/.emacs.d= directly). It finds the newest =.pre-toolkit-<STAMP>= set, archives the current post-install state as =.rolled-back-<NOW>=, and swaps the backups back into place. The backups remain on disk until you delete them; rollback is always reversible.

Launch Emacs and check =*Messages*=:

#+begin_example
my/offline-packages=t  dir=/home/you/elpa-mirror-emacs-...
#+end_example

If =dir=<none found>= appears, the mirror wasn't extracted into =$HOME= or the =~/elpa-mirror-emacs-*= glob didn't match — see Troubleshooting.

* How the online/offline toggle works

The rendered =init.el= sets:

#+begin_src emacs-lisp
(setq my/offline-packages t)     ; set to nil to force online archives
(setq my/offline-packages-dir
      (car (sort (file-expand-wildcards
                  (expand-file-name "~/elpa-mirror-emacs-*"))
                 #'string>)))
#+end_src

- =t= + a discovered mirror dir → =package-archives= is set to the local mirror only.
- =nil=, or no mirror found → =package-archives= falls back to MELPA / ELPA / Org ELPA over the network.

The mirror directory is detected automatically by globbing =~/elpa-mirror-emacs-*=; newest wins. No edits needed when a new mirror is installed. The same file works:

- at =create-install.sh= time (no mirror in the staged HOME → online archives used to download packages), and
- on the deployed offline target (mirror extracted into =$HOME= → local archive used).

All per-package =require= calls are wrapped =(require 'foo nil t)= so a missing package logs rather than aborts. =custom-set-faces= is wrapped in =condition-case= for the same reason.

* Pre-fetching GNU Emacs source

=fetch-source.sh= downloads a GNU Emacs source tarball into =sources/=:

#+begin_src sh
./fetch-source.sh          # reads sources/LATEST_STABLE (e.g. "30.1")
./fetch-source.sh 27.2     # explicit version
#+end_src

The download is idempotent — if =sources/emacs-<VER>.tar.xz= already exists the script is a no-op. Tarballs are gitignored.

=build-emacs-versions.sh= picks them up automatically when =create-install.sh= invokes it: the orchestrator exports =SOURCES_DIR="$SCRIPT_DIR/sources"=, and the build script prefers a tarball there over a fresh =wget= from =ftp.gnu.org=. Pre-fetch once on the online machine and subsequent builds (online or offline-then-online) reuse the cached source.

=build-toolkit.sh --with-source auto= (opt-in; pass the flag explicitly, or pass it through =create-install.sh --with-source auto=) reads the version from =sources/LATEST_STABLE= and bundles =sources/emacs-<LATEST_STABLE>.tar.xz=. If the tarball isn't cached it downloads once from =ftpmirror.gnu.org= into =sources/=. One canonical source version is bundled across /all/ target mirrors.

* Rebuilding Emacs from the bundled source (optional)

When =--with-source VER= (or =--with-source auto=) was used at toolkit build time, an =emacs-<VER>.tar.xz= is bundled. To build:

#+begin_src sh
tar -xf emacs-<VER>.tar.xz -C /tmp
cd /tmp/emacs-<VER>
./configure --prefix="$HOME/.local"
make -j$(nproc)
make install
export PATH="$HOME/.local/bin:$PATH"
#+end_src

Dependencies the =configure= step will typically want from the distro's /own/ offline repos:

- =libgtk-3-dev= or =libncurses-dev= (GUI vs TUI)
- =libgnutls28-dev=, =libxml2-dev=, =libjpeg-dev=, =libpng-dev=, =libtiff-dev=
- =libgif-dev=, =libjansson-dev=, =libtree-sitter-dev=

These are outside the toolkit — each distro bundles them differently.

For building /many/ Emacs versions on the online machine, see =Emacs-vanilla/scripts/build-emacs-versions.sh=, which =create-install.sh= invokes automatically if the requested version isn't already installed.

* Troubleshooting

** Package X skipped at startup

Look in =*Messages*= for:

#+begin_example
my/ensure-package: skipping ef-themes ((error . "..."))
#+end_example

Reasons:
- Package requires a newer Emacs than the target (=ef-themes= needs 28.1+).
- Package wasn't in =packages/emacs-<VER>.el= when the mirror was built — add it and re-run =create-install.sh=.

** my/offline-packages-dir=<none found>

The ELPA mirror didn't get extracted into =$HOME=. Run:

#+begin_src sh
ls -d ~/elpa-mirror-emacs-*
#+end_src

If nothing matches, re-run =setup.sh= from inside the extracted toolkit or extract the mirror manually:

#+begin_src sh
tar -xzf ~/.../elpa-mirror-emacs-*.tar.gz -C "$HOME"
#+end_src

** Mixed Emacs versions

Mirrors are version-locked. Don't build a mirror on Emacs 30 and deploy it to Emacs 27 — some packages (e.g. =transient=) will be rejected by =package-install=. =create-install.sh= enforces this by running the install under a specifically named =emacs-<VER>= binary.
